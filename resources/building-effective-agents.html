🤖 构建高效的 AI 智能体

Anthropic 官方指南中文版 — 基于数十个团队落地经验的权威实践建议

📌 核心观点（先读这个）

最成功的智能体实现，使用的都是简单、可组合的模式，而不是复杂的框架。这篇文章会教你五种经过生产验证的工作流模式，以及何时该用智能体、何时不该用。

一、什么是"智能体"？

"智能体"这个词有几种不同的定义。有些客户把智能体定义为完全自主的系统——能够长期独立运行，使用各种工具完成复杂任务。另一些则用这个词描述遵循预定义工作流的更规范的实现。

在 Anthropic，我们把所有这些变体都归类为智能体系统（agentic systems），但在架构上做了一个重要区分：

⚙️ 工作流（Workflow）

LLM 和工具通过预定义的代码路径来编排。每一步都写死了，执行顺序是固定的。

🤖 智能体（Agent）

LLM动态控制自己的流程和工具使用，自主决定如何完成任务。

二、何时（以及何时不）使用智能体

用 LLM 构建应用时，我们建议尽可能找到最简单的解决方案，只在必要时才增加复杂度。这可能意味着根本不需要构建智能体系统。

⚠️ 重要权衡

智能体系统通常用延迟和成本来换取更好的任务性能。你必须想清楚这个权衡是否值得。

当需要更多复杂度时：

  • 工作流适合明确定义的任务——提供可预测性和一致性
  • 智能体适合需要灵活性和模型驱动决策的场景——在大规模下更优
  • 对许多应用来说，优化单个 LLM 调用（加上检索和上下文示例）通常就够用了

三、何时以及如何使用框架

有很多框架可以让智能体系统更容易实现，包括：

  • Claude Agent SDK — Anthropic 官方 Agent SDK
  • Strands Agents SDK (AWS) — AWS 的 Agent 框架
  • Rivet — 拖拽式 GUI LLM 工作流构建器
  • Vellum — 用于构建和测试复杂工作流的 GUI 工具
💡 Anthropic 的建议
这些框架通过简化标准底层任务（调用 LLM、定义和解析工具、链式调用）让入门变得容易。但它们往往创建了额外的抽象层，可能掩盖底层的提示和响应，让调试变得更困难。开发者应该先用 LLM API 直接开发：许多模式几行代码就能实现。如果用了框架，也要确保理解底层代码。

四、构建模块、工作流与智能体

下面我们深入探讨在生产环境中见过的常见智能体系统模式，从基础构建模块开始，逐步增加复杂度。

🔧 基础构建模块：增强型 LLM

智能体系统的基本构建模块是一个增强了检索、工具和记忆能力的 LLM。当前模型可以主动使用这些能力——生成自己的搜索查询、选择合适的工具、决定保留哪些信息。

⚙️ 工作流模式一：提示链（Prompt Chaining）

提示链将一个任务分解为一系列步骤，每个 LLM 调用处理前一个的输出。可以在任何中间步骤添加程序化检查（"门控"），确保流程仍在正轨上。

✅ 何时使用：任务可以清晰、干净地分解为固定子任务时。主要目标是通过让每个 LLM 调用处理更简单的任务，以延迟换取更高准确性。

例子：生成营销文案 → 翻译成不同语言；写文档大纲 → 检查大纲是否符合标准 → 根据大纲写完整文档。

⚙️ 工作流模式二：路由（Routing）

路由对输入进行分类，并将其导向专门的后续任务。这种工作流允许关注点分离，构建更专门的提示词。

✅ 何时使用：有不同类别需要分别处理，且分类可以由 LLM 或更传统的分类模型准确处理时。

例子：将不同类型的客户支持查询（一般问题、退款请求、技术支持）路由到不同的下游流程；将简单问题路由到低成本模型（如 Claude Haiku），困难问题路由到更强模型（如 Claude Sonnet）。

⚙️ 工作流模式三：并行化（Parallelization）

LLM 可以同时处理任务，然后以编程方式聚合输出。并行化有两种主要变体：

  • 分片（Sectioning）：将任务分解为独立子任务并行运行
  • 投票（Voting）：多次运行同一任务以获取多样化输出
✅ 何时使用：子任务可以并行化以提速，或需要多个视角/多次尝试以提高结果可信度时。

例子（分片）：一个模型实例处理用户查询，另一个筛查不当内容；用 LLM 评估模型性能的不同方面。
例子（投票）：用多个不同提示审查代码漏洞；用多个提示评估内容是否不当。

⚙️ 工作流模式四：编排器-工作器（Orchestrator-Workers）

中央 LLM 动态分解任务，委派给工作 LLM，然后综合它们的结果。

✅ 何时使用：无法预测需要哪些子任务的复杂任务（例如编程中需要变更的文件数量和每个文件的变更性质取决于具体任务）。与并行化的关键区别是灵活性——子任务不是预定义的，而是由编排器根据特定输入决定。

例子：每次都对多个文件进行复杂变更的编程产品；需要从多个来源收集和分析信息的搜索任务。

⚙️ 工作流模式五：评估器-优化器（Evaluator-Optimizer）

一个 LLM 调用生成响应，另一个在循环中提供评估与反馈。

✅ 何时使用：有清晰评估标准，且迭代改进能带来可衡量价值时。两个适配信号：① 人类明确表达反馈后 LLM 响应可以显著改进；② LLM 可以提供此类反馈。

例子：文学翻译（有细微差别初始翻译可能捕捉不到）；需要多轮搜索和分析的复杂搜索任务。

🤖 智能体（Agents）

随着 LLM 在关键能力上成熟——理解复杂输入、进行推理和规划、可靠使用工具、从错误中恢复——智能体正在生产环境中涌现。

智能体从人类用户的命令或交互讨论开始工作。任务明确后，智能体自主规划和操作，可能返回人类处获取更多信息或判断。执行过程中，智能体在每一步从环境获取"真值"（如工具调用结果或代码执行结果）以评估进展至关重要。

✅ 何时使用智能体
适用于难以或无法预测所需步骤数量的开放式问题，以及无法硬编码固定路径的场景。LLM 可能运行多轮，你必须对其决策有一定程度的信任。智能体的自主性使其非常适合在受信任环境中扩展任务。

⚠️ 注意：自主性意味着更高成本和潜在错误累积。建议在沙箱环境中广泛测试，并配备适当的防护措施。

智能体实例：解决 SWE-bench 编程任务的编码智能体；Anthropic 的"computer use"参考实现（Claude 使用计算机完成任务）。

五、组合和定制这些模式

这些构建模块不是规定性的。它们是开发者可以根据不同用例塑造和组合的常见模式。成功的关键是——与任何 LLM 功能一样——衡量性能并迭代实现。

🎯 记住：只有在证明能改善结果时才增加复杂度。

六、总结：三个核心原则

LLM 领域的成功不在于构建最复杂的系统，而在于为你的需求构建正确的系统。

1. 保持简单

从简单提示开始，通过全面评估优化它们，只在更简单的解决方案不够用时才添加多步智能体系统。

2. 优先考虑透明度

通过明确展示智能体的规划步骤，让决策过程透明可查。

3. 精心设计工具接口（ACI）

通过彻底的工具文档和测试，精心设计智能体-计算机接口。这和产品管理中的 HCI（人机界面）一样重要。

附录一：智能体的实际应用场景

A. 客户支持

客户支持将熟悉的聊天机器人界面与工具集成增强的能力相结合。这是开放式智能体的天然契合场景：支持交互自然遵循对话流程，同时需要访问外部信息和操作；可以集成工具来拉取客户数据、订单历史和知识库文章；退款或更新工单等操作可以程序化处理；成功可以通过用户定义的解决方案清晰衡量。

B. 编码智能体

软件开发领域展现了 LLM 功能的卓越潜力。智能体特别有效，因为：代码解决方案可以通过自动化测试验证；智能体可以使用测试结果作为反馈来迭代解决方案；问题空间定义良好且结构化；输出质量可以客观衡量。

附录二：给你的工具做提示工程

无论构建哪种智能体系统，工具都是重要组成部分。工具定义和规范应该像整体提示词一样接受提示工程关注。

💡 工具格式设计建议
  • 给模型足够的 token 来"思考"，避免把它逼入死角
  • 保持格式接近模型在互联网文本中自然出现的样子
  • 确保没有格式"开销"——比如需要准确计算数千行代码，或对写的任何代码进行字符串转义
  • 像设计人机界面（HCI）一样，投入同样多的精力创建好的智能体-计算机界面（ACI）
