🔍 RAG 检索增强生成

给 PM 的 RAG 实用指南 — 理解原理、评估需求、设计产品

📌 一句话解释 RAG：RAG = 让 LLM 在回答问题时，先去查你的专有资料，再根据查到的内容回答。这样既利用了 LLM 的语言能力，又避免了它"编造"答案。

一、为什么需要 RAG？

直接用 LLM（比如把公司文档全部贴进提示词）有三个问题：

  1. 上下文窗口有限：你塞不进几百份文档
  2. 成本高：每次对话都要把文档内容发一遍，按 token 收费
  3. 信息过时：LLM 训练完就不再更新知识了，它不知道你公司上周发生的事

RAG 就是为了解决这三个问题而生的。

二、RAG 是怎么工作的？

预处理阶段（离线）

📄 你的文档 → ✂️ 切成片段 → 🔢 转成向量（Embedding） → 💾 存入向量数据库

查询阶段（在线）

❓ 用户提问 → 🔢 问题转成向量 → 🔍 在向量数据库找最相似的文档片段 → 🤖 LLM 根据检索到的片段回答
💡 关键理解：RAG 不是把"所有知识"塞进 LLM，而是在回答每个问题时，只检索最相关的那几段。就像开卷考试——你不需要背整本书，只需要知道去哪一页找答案。

三、RAG 的详细步骤（PM 视角）

Step 1：文档切片（Chunking）

把文档切成小片段（通常 500-1000 字符）。这看起来简单，但很关键：

  • 切太大 → 噪音多，检索不精准
  • 切太小 → 丢失上下文，回答不完整
  • 产品决策：你的文档是什么类型？客服 FAQ 适合按 Q&A 对切；技术文档适合按章节切；PDF 要注意表格和图片的处理

Step 2：向量化（Embedding）

用 Embedding 模型把文字转成数字向量（一串几百到几千维的数字）。语义相近的文字，向量也相近。

常用模型：OpenAI 的 text-embedding-3-small（便宜、好用）、Cohere Embed、开源的 BGE 模型。

Step 3：存储（Vector Database）

把向量存起来，支持"根据向量相似度搜索"。常用数据库：

  • Pinecone：托管服务，最简单，免费版可用
  • Chroma：开源，本地运行，适合 prototyping
  • Weaviate / Qdrant：开源，功能更强，适合生产环境

Step 4：检索（Retrieval）

用户提问 → 转成向量 → 在数据库中找最相似的 Top-K 个片段（通常 K=3~10）。

PM 要注意：检索质量直接决定回答质量。如果检索出来的片段不相关，LLM 再强也回答不好。这叫做"垃圾进，垃圾出"。

Step 5：生成（Generation）

把检索到的片段 + 用户问题一起发给 LLM，让它基于片段回答。关键提示技巧：告诉 LLM "如果参考资料中没有相关信息，就说不知道，不要编造"。

四、RAG 的进化版本

基础版 RAG（Vanilla RAG）效果有限。生产环境中常用这些改进版本：

🔄 自适应 RAG（Adaptive RAG）

根据问题类型动态选择检索策略。简单问题少检索，复杂问题多检索。也可以用 LLM 先"改写"用户问题，提高检索准确率。

🔀 混合 RAG（Hybrid RAG）

结合关键词搜索（BM25）+ 语义搜索（向量检索），取长补短。关键词搜索擅长精确匹配，语义搜索擅长理解意图。

🤔 纠正性 RAG（CRAG）

检索后先评估检索结果的质量，如果质量低就去网络搜索补充信息。相当于给 RAG 加了一个"自我怀疑"机制。

📚 图 RAG（Graph RAG）

用知识图谱增强 RAG，捕捉实体之间的关系。适合需要处理复杂关联关系的场景（比如"某公司的供应链上下游有哪些企业"）。微软开源了 GraphRAG 框架。

五、PM 必备：RAG 产品的评估框架

✅ 你的产品需要 RAG 吗？

问自己：LLM 的通用知识够用吗？需要引用来源吗？知识会更新吗？如果三个都是"是"，你需要 RAG。

✅ 怎么评估 RAG 系统质量？

  • 检索准确率：检索出来的片段是否相关？（人工评估 or LLM 评估）
  • 回答准确率：最终回答是否正确？（需要标准答案数据集）
  • 拒绝率：当资料中没有答案时，系统是否会拒绝回答（而不是编造）？

✅ 常见坑

  • 文档切片策略不当，导致上下文丢失
  • 检索结果没有按相关性排序，最相关的片段排在后面
  • 没有告诉 LLM "不知道就说不知道"，导致幻觉
  • 向量数据库没有更新机制，过时信息一直存在
