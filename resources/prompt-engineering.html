✍️ 提示工程完全指南

Anthropic 官方最佳实践中文版 — 给 PM 和开发者的实用提示词技巧

🏆 黄金法则（必读）

"把你的提示词展示给一位对任务了解有限的同事，请他们按照提示词操作。如果他们会感到困惑，Claude 也会如此。"

把 Claude 当作一位聪明但缺乏经验的新员工——它不了解你的规范和工作流程。你越精确地解释你想要什么，结果就越好。

一、通用原则（适用于所有 Claude 模型）

基础技巧

1. 清晰直接地表达

Claude 对清晰、明确的指令响应良好。明确说明你期望的输出有助于提升结果质量。

  • 明确说明期望的输出格式和约束条件
  • 当步骤顺序很重要时，使用编号列表提供指令
  • 避免让 Claude 从模糊提示中推断你的意图
基础技巧

2. 添加上下文以提升性能

在指令背后提供上下文或动机（比如解释为什么某种行为很重要），可以帮助 Claude 更好地理解你的目标并提供更有针对性的响应。

你是一位有 10 年经验的产品经理，正在为一款面向中老年用户的健康 App 写功能说明。请用简单易懂的语言解释「心率监测」功能，避免专业术语。
核心技巧

3. 有效使用示例（Few-shot Prompting）

示例是引导 Claude 输出格式、语气和结构的最可靠方法之一。包含 3-5 个示例可获得最佳效果。

  • 相关性：尽可能贴近实际使用场景
  • 多样性：覆盖边缘情况
  • 结构化：将示例包裹在 <example> 标签中
核心技巧

4. 使用 XML 标签结构化提示词

XML 标签帮助 Claude 清晰解析复杂提示词，特别是当提示词混合了指令、上下文、示例和可变输入时。

<instructions> 你是一个客服助手。根据用户问题提供有帮助的回答。 </instructions> <context> 我们的退款政策是：收到商品后 30 天内可申请退款。 </context> <input> 用户问题：我上周买的东西可以退货吗？ </input>
进阶技巧

5. 给 Claude 一个角色

在系统提示词中设置角色可以使 Claude 的行为和语气聚焦于你的使用场景。即使是一句话也能产生差异。

system: "你是一位专业的产品策略顾问，擅长用数据和用户洞察来支持产品决策。"

二、控制输出格式

有几种特别有效的方法来引导输出格式：

✅ 方法一：告诉 Claude 要做什么，而不是不要做什么

不要用否定式指令，用正面指令：

  • ❌ "Do not use markdown in your response"
  • ✅ "Your response should be composed of smoothly flowing prose paragraphs."

✅ 方法二：使用 XML 格式指示符

把你的回复写在 <smoothly_flowing_prose_paragraphs> 标签中。

✅ 方法三：使提示词风格与期望输出相匹配

你提示词中使用的格式风格可能影响 Claude 的响应风格。如果你想减少输出中的 markdown，试着从提示词中删除 markdown。

三、利用思考（Thinking）能力

Claude 最新模型提供思考能力（Thinking/Reasoning），对于涉及工具使用后的反思或复杂多步推理任务特别有帮助。

💡 什么是思考（Thinking）？
思考是 Claude 在回答之前进行的内部推理过程。就像你在回答难题前先在脑子里想一遍。你可以看到 Claude 的思考过程（在 API 中通过 thinking 参数开启）。

如何引导 Claude 的思考行为：

  • 在收到工具结果后，要求 Claude "仔细反思其质量，然后确定最佳下一步"
  • 使用自适应思考（thinking: {type: "adaptive"}），Claude 会根据查询复杂度自动决定思考多少
  • 要求 Claude 自我检查："在结束之前，根据 [测试标准] 验证你的答案"

四、智能体系统提示技巧

🤖 长时间推理和状态追踪

Claude 处理长时间推理任务时具有强大的状态追踪能力。关键技巧：

  • 要求 Claude 在继续之前将进度保存到文件
  • 使用结构化格式（如 tests.json）跟踪测试结果
  • 使用 git 进行状态追踪——Claude 在使用 git 跨多个会话追踪状态方面表现出色

🤖 平衡自主性和安全性

如果没有指导，Claude 可能会采取难以逆转的操作（删除文件、强制推送等）。添加安全指导：

<safety_guidelines> 对于难以逆转的操作（删除文件、git push --force、影响共享系统的操作）， 在执行前先询问用户确认。 </safety_guidelines>

🤖 避免过度工程化

Claude Opus 4.5+ 倾向于过度工程化。添加指导保持解决方案简单：

<minimize_over_engineering> - 范围：不要添加未被请求的功能或"改进" - 文档：不要为你未更改的代码添加注释 - 抽象：不要为一次性操作创建辅助函数 - 防御性编程：不要为不可能发生的场景添加错误处理 </minimize_over_engineering>

五、PM 常用提示词模板

📋 模板一：需求分析

你是一位资深产品经理。请分析以下用户反馈，提取核心需求， 按优先级排序，并给出产品建议。 用户反馈： [粘贴用户反馈 here] 输出格式： 1. 核心需求（按优先级） 2. 每个需求的影响范围和频率 3. 推荐的产品方案 4. 优先级排序理由

📋 模板二：竞品分析

你是一位产品战略顾问。请对以下产品进行竞品分析。 产品 A：[产品名] - [核心功能] 产品 B：[产品名] - [核心功能] 请从以下维度分析： - 目标用户差异 - 核心功能差异 - 用户体验差异 - 商业模式差异 - 各自的竞争优势

📋 模板三：AI 功能设计

你正在为 [产品名] 设计一个 AI 功能：[功能描述] 请帮助我思考： 1. 这个功能是应该用机器学习、深度学习还是生成式 AI？为什么？ 2. 用户价值是什么？能不能用更简单的方法实现？ 3. 成功指标应该怎么定义？ 4. 有哪些风险和边界情况需要考虑？
